    SQL Injections: The Bigger Picture



    What are SQL queries:

    SQL is a standardized language used to maintain and manipulate databases to create a custom query specific to what a user needs. These queries are used to execute commands such as data retrieval, data removal and updating what’s currently in the database. Different SQL elements implement these tasks, such as SELECT, WHERE, RETURN, and MATCHING. We can further filter the data with conditional statements or keywords if the results are to broad.



    This is what a typical query may look like. In this example we get the items name and description from the table: Item. 



                        
                        SELECT ItemName, ItemDescription
                        FROM Item
                        WHERE ItemNumber = ItemNumber


    Looks promising right? Well from here, the web application builds a string query that is then sent to the database to retrieve your data.



                        
                        //Converted query for SQL to read
                        sql_query= "
                        SELECT ItemName, ItemDescription
                        FROM Item
                        WHERE ItemNumber = " & Request.QueryString("ItemID")


    But you ask yourself how do we know which item we are referring to? Well this is where we can ask the user for input or have pre-made guided solutions to this problem as to not have a user return something they shouldn't have access to. We then make sure to match the item number given to the one already in the database. The final step is taking the provided users input such as itemid=999 to generate the final query. 



                        
                        
                        SELECT ItemName, ItemDescription
                        FROM Item
                        //users input
                        WHERE ItemNumber = 999


    As you probably guessed already it will return the data on the specific item only in this example and nothing else.



    Item # Name Description

    One thing to note is that we need to make sure each item in a table has a unique identifier so that we can reference it later. This also should not be editable as changing this identifier can cause issues down the line. Instead if we wanted to update or delete an item we can edit the other contents of a specific item in the table.

    The problem with this approach is that there is no way to know what the user will put in here making it possible for attacks!     Safe APIs:

    What are Safe APIs?

    With XSS-prone APIs identified and forbidden in the development cycle, Google needed to come up with a set of "Hardened" APIs not known to be vulnerable. In order for a API to be considered "Safe" it has to have met the three critera first:

    1. Safe Type
    2. Safe Builders
    3. Safe Sinks
    What are Safe Types?

    There are a total of six safe types to cover a majority of attacks known in JavaScript and DOM manipulation. Each safe type corresponds to a type of sink. Values of safe types are immutable once created and are better known as the wrappers of primative strings. The internals of safe types are hidden from developers and in TypeScript, we mark them as private properties. In Javascript, Google implemented additional compile-time checks to forbid any code from accessing these private properties. These properties also prevent safe types from being confuesed with other user-made types.

    What are Safe Builders?

    Builders of safe values are essentially the core security checkpoint for the API Hardening technique. Generally, safe values are secure from sensative DOM sinks due to the static checks in place and APIs ensure they can only be created in ways known to be secure. There are a total of four ways to constuct safe values:

    Build from literal values: These values are hard coded by developers and cannot be manipulated by attackers. These can then be safely converted to safe type. One note to know is that literal values are not inhearently secure, only by making the proper checks and making it secure is it safe.

    Build by sanitizing: Google created built in sanitizers to automatically fix untrustworthy strings into safe-to-render HTML markup or safe-to-follow URLs, removing the known XSS vunerability.

    Build from template: Building from a template can be seen as a combination of building from constants and building from sanitizing. The bulk of the template is static while some parts can be configured with run-time values.

    Build from manually reviewed sources: If all else fails and none of the previous options listed before suit your need a developer can use this last method to case any value as a safe value. This is a backdoor technique in the safe API design, however they enforce review requirements by making the developer classify them as prohibited XSS sinks and add them to the exemption list.

    What are Safe Sinks?

    The accompanying image is an example of an original version of an API and the new Hardened version of a Safe sink in TypeScript. In hindsight, safe sinks are the hardened version of XSS-prone JavaScript and DOM APIs. Typically, safe sinks are type-safe wrappers of the original APIs. If this sink does not have a safe-type associated with the values given at the time of execution it will be void instead and not run.
